# Use Tempo access keys with MPP

Access keys let a Tempo wallet pay in the background without prompting for every request. Use them when a payment-aware client needs a bounded budget.

Use access keys to:

- cap spend per token and period
- restrict payments to specific contracts, functions, or recipients
- revoke one app or workflow without changing the wallet

## Connect with a payment key

Use your existing Tempo Wallet provider. This example creates one for a browser app.

```ts [wallet.ts]
import { Expiry, Provider } from 'accounts'
import { parseUnits } from 'viem'

const usdc = '0x20C000000000000000000000b9537d11c60E8b50' as const

export const provider = Provider.create()

const paymentKey = {
  expiry: Expiry.days(7),
  limits: [{ token: usdc, limit: parseUnits('10', 6), period: 86_400 }],
}

await provider.request({
  method: 'wallet_connect',
  params: [{ capabilities: { authorizeAccessKey: paymentKey } }],
})
```

This creates a seven-day key with a 10 USDC daily budget. Add `scopes` when a runtime should only call specific contracts or functions. For example, scope a key to USDC transfers for one merchant so the runtime cannot spend that allowance anywhere else.

If the wallet is already connected, call `wallet_authorizeAccessKey` with the same `paymentKey`.

## Make fetch payment-aware

Pass the Tempo account to `mppx`.

```ts [payments.ts]
import { Mppx, tempo } from 'mppx/client'
import { provider } from './wallet'

Mppx.create({
  methods: [
    tempo({
      account: provider.getAccount(),
      ...provider.getMppxParameters(),
    }),
  ],
})
```

Use `fetch` normally for unpaid requests, paid HTTP requests, and MCP transports that accept `fetch`. `mppx` only pays when the server returns a payment challenge, and the wallet uses a matching access key when one is available.

Create separate access keys for separate apps, tools, or deployments. That keeps each payment surface independently revocable and gives every runtime its own expiry, budget, and scope.

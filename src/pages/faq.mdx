# Frequently asked questions [Common questions about the Machine Payments Protocol]

## General

### MPP overview

The Machine Payments Protocol (MPP) is an internet-native protocol for machine-to-machine payments. It standardizes HTTP `402` "Payment Required" with an extensible framework that works with any payment network. Clients pay inline with their requests, and servers receive payment confirmation before delivering the response.

### Comparison to existing payment solutions

Traditional payment solutions require manual signup, API keys, billing accounts, or browser-based checkout flows. MPP eliminates these requirements by embedding payment negotiation directly in HTTP headers. The server sends a challenge describing what payment is needed, the client fulfills it, and retries the request with a credential.

### Production readiness

MPP is in active development. The [Tempo payment method](/payment-methods/tempo) is production-ready for stablecoin payments. Other payment methods, including Stripe, are in beta.

## Technical

### Supported payment methods

MPP supports multiple payment methods through its extensible design. Currently available methods include:

- **Tempo**—Native stablecoin payments on the Tempo blockchain (production)
- **Stripe**—Traditional card and bank payments via Stripe (beta)

Anyone can define new payment methods. See the [Payment Methods](/payment-methods) page for details.

### Integrating with existing infrastructure

MPP works with any HTTP server or framework. The protocol uses standard HTTP headers (`WWW-Authenticate`, `Authorization`, `Payment-Receipt`) and status codes (`402`). Use the `mpay` SDK for TypeScript, Python, or Rust to add payment support to your existing APIs.

### Payment verification

When a client submits a credential, the server validates the payment proof against the original challenge. The verification process depends on the payment method—for example, the Tempo method verifies that the client signed a valid transaction. After verification, the server settles the payment and returns a receipt.

## Security

### Security model

MPP requires TLS 1.2 or later for all payment flows. Payment methods must provide single-use proof semantics, preventing replay attacks. Challenges are cryptographically bound to their parameters, ensuring clients can't reuse a challenge ID with modified payment terms. See [Security considerations](/protocol#security-considerations) for details.

### Handling failed payments

When a payment fails, the server returns `402` with a fresh challenge and a Problem Details body describing the error. Common failure reasons include expired challenges, invalid payment proofs, or insufficient payment amounts. Clients can retry with a new credential based on the fresh challenge.
